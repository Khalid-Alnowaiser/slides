Automated Generation of SSL Certificates using Certbot / LetsEncrypt

Option 1- http-01 challenge

Pros & Cons

  • Pros
    • Easier to setup
    • Customer points 1 CNAME record only
  • Cons
    • Port 80 has to be open
    • No wildcard domains
    • Challenge file must be placed on all servers

Option 2- dns-01 challenge

DNS challenge process workflow

Process workflow explanation

To streamline this process, we create a “validation zone” that the customer will point to using 

_acme-challenge.lucidya.khalidalnowaiser.tech

When Let’s Encrypt checks the TXT records for 

_acme-challenge.lucidya.khalidalnowaiser.tech
it will be redirected to 
_acme-challenge.lucidya.khalidalnowaiser.alnowaiser.me
which contains the needed TXT record value.

Pros & Cons

  • Pros
    • Supports wildcard domains
    • Works well even if we have multiple web servers
    • Can be used even if port 80 is blocked
  • Cons
    • More complex to setup from our side
    • Customer sets an extra CNAME records
    • Need to use our Cloudflare API key for TXT records

dns-01 challenge config example

My setup:

  • Ubuntu 24.04 (LTS) x64
  • nginx version: nginx/1.24.0 (Ubuntu)
  • certbot 2.11.0 with Cloudflare DNS plugin installed

Prerequisites:

  • API key with access to edit DNS settings
  • Setup config file

Setup:

I’ve setup an SSL for alnowaiser.me

Customer setup:

Customer (lucidya.khalidalnowaiser.tech) sets up 2 CNAME records:


Name: lucidya.alnowaiser.tech
Value: alnowaiser.me


Name: _acme-challenge.lucidya.khalidalnowaiser.tech
Value: _acme-challenge.lucidya.khalidalnowaiser.alnowaiser.me

Generating TXT record:

After the customer adds the CNAME records, we can generate the TXT record under our own domain 

_acme-challenge.lucidya.khalidalnowaiser.alnowaiser.me
by using:


certbot certonly --dns-cloudflare --dns-cloudflare-credentials (path to cloudflare.ini)/cloudflare.ini -d lucidya.khalidalnowaiser.tech -d *.lucidya.khalidalnowaiser.tech (optional)

Success

Setup webserver config:

In my case, I’m using nginx (I did it manually, but this can be automated)

Afterwards, we have to reload the config for the changes to take effect

Why Use This Approach?

Since we’re using Let’s Encrypt for our SSL certificates, which expire every 90 days, we need to renew them approximately every 60 days to ensure continuous security.

Let’s Encrypt uses TXT records to verify domain ownership, and these records must be updated each time the SSL certificate is renewed. This means the customer would need to manually update the TXT record values every 60 days, which is cumbersome and prone to error.

Benefits:

By using this approach, we eliminate the need for customers to manually update their TXT records every 60 days.

Potential Issues

  • CNAME Cross-User Banned
    • Potential Fix: Ensure that the user's cname record isn't proxied if they're using cloudflare
  • Domain Forwarding
  • Other security / lucidya-specific issues
